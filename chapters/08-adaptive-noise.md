# 第八章：自适应检索与噪声过滤

> 不是所有对话都值得检索记忆。当用户说"你好"、输入一个 emoji、或者敲下 `/clear` 命令时，触发一次完整的混合检索管线是纯粹的资源浪费。更糟糕的是，这些无意义的查询可能召回不相关的记忆，干扰 LLM 的上下文。memory-lancedb-pro 用三道防线解决这个问题：自适应检索在入口处拦截无效查询，噪声过滤在出口处清除低质量结果，噪声原型库则用 embedding 的方式实现语言无关的噪声识别与自学习。

## 8.1 自适应检索：入口处的守门员

`adaptive-retrieval.ts` 是一个精巧的 100 行模块，它在检索管线的最前端做出三种判断：跳过（skip）、强制触发（force）、或正常通过（normal）。

```
// 文件: src/adaptive-retrieval.ts L10-30
// 查询分类逻辑：
// 1. 先检查是否应该跳过（skip）
// 2. 再检查是否应该强制检索（force）
// 3. 最后检查最小长度要求
// 4. 通过所有检查 → normal（正常进入管线）
```

**为什么跳过检查在强制检查之前？** 这个优先级设计是经过深思熟虑的。考虑一个边界情况：用户输入 "/remember"——它同时匹配命令模式（以 / 开头）和强制触发关键词（"remember"）。如果先检查强制触发，这个明显的命令输入就会触发一次检索。通过将跳过检查放在首位，我们确保了明确的无效输入不会被后续规则"误救"。

### 核心流程

```
三层噪声防护协同流程:

第一层: adaptive-retrieval.ts (入口守门, 微秒级)
+--------+    +-------+    +-------+    +--------+
| 输入    +--->+ Skip  +-N->+ Force +-N->+ MinLen +--> normal
| 查询    |    | 检查   |    | 检查   |    | 检查    |    (进入管线)
+--------+    +---+---+    +---+---+    +---+----+
                  |Y           |Y           |Y(太短)
                  v            v            v
               [跳过]      [强制检索]     [跳过]

第二层: noise-filter.ts (出口质检, 微秒级)
+----------+    +----------+    +----------+    +---------+
| 检索结果  +--->+ Agent    +--->+ 元问题    +--->+ 样板    +--> 干净结果
| 候选集    |    | 拒绝模式  |    | 模式     |    | 文本    |
+----------+    | (15 种)   |    | (20+ 种) |    | 模式    |
               +----------+    +----------+    +---------+

第三层: noise-prototypes.ts (语义检测, 毫秒级)
+----------+    +-----------+    +---------+    +---------+
| 候选记忆  +--->+ 15 种子    +--->+ 余弦    +--->+ 超阈值?  |
|           |    | 原型向量   |    | 相似度   |    | → 噪声  |
+----------+    +-----------+    +---------+    +----+----+
                     ^                               |
                     |                          [零提取信号]
                +----+----+                          |
                | 自学习   +<-------------------------+
                | 新原型加入|  (去重阈值 0.95, 上限 200)
                +---------+
```

### Skip 规则

```
// 文件: src/adaptive-retrieval.ts L35-55
// 跳过条件（任一满足即跳过）：
// 1. 问候语: hi, hello, hey, 你好, こんにちは 等
// 2. 命令: 以 / 开头的文本（如 /new, /clear, /help）
// 3. 肯定回应: ok, yes, sure, 好的, 没问题 等
// 4. 纯 emoji: 全部由 emoji 字符组成的消息
```

这四类 skip 规则覆盖了对话中最常见的"非信息性"输入。它们的共同特征是：不包含任何可供检索的语义内容。对这类输入执行检索，最好的情况是浪费计算资源返回空结果，最坏的情况是召回不相关的记忆干扰后续对话。

**问候语识别为什么需要多语言支持？** memory-lancedb-pro 定位为全球可用的记忆插件。一个只能识别英文 "hello" 但无法识别中文"你好"或日文"こんにちは"的跳过规则，在非英语环境中形同虚设。多语言问候语列表虽然增加了维护成本，但对系统的实际可用性至关重要。

### Force 规则

```
// 文件: src/adaptive-retrieval.ts L60-80
// 强制检索条件（任一满足即强制触发）：
// 1. 英文记忆关键词: "remember", "recall", "last time",
//    "previously", "we discussed" 等
// 2. CJK 记忆关键词: "记得", "上次", "之前", "提到过" 等
// 3. 这些关键词明确表达了用户对历史记忆的检索意图
```

Force 规则的存在解决了一个重要问题：某些查询可能因为太短而被最小长度规则过滤掉，但它们包含明确的检索意图。例如，用户说"上次呢？"——只有 4 个中文字符，低于很多长度阈值，但"上次"这个词明确表达了回忆历史信息的意图。Force 规则确保了这类查询不会被误过滤。

### 最小长度要求

```
// 文件: src/adaptive-retrieval.ts L85-100
// 最小长度阈值：
// 英文: 15 个字符
// CJK: 6 个字符
//
// 为什么 CJK 阈值更低？
// 因为 CJK 文字的信息密度远高于拉丁字母
// "数据库部署方案" = 6 个字符，但信息量等同于
// "database deployment plan" = 24 个字符
```

这个看似简单的长度检查背后是对语言特性的理解。中文一个字就是一个语素，而英文一个单词通常需要 4-7 个字符。如果对 CJK 文本使用与英文相同的字符数阈值，大量有效的中文查询都会被错误地跳过。6 个中文字符大约是 2-3 个词，足以表达一个有意义的检索意图。

### 设计取舍

自适应检索的 skip/force/normal 三路分类看似简单，实际上否定了两个替代方案。第一个是**纯长度过滤**——只按字符数决定是否跳过。这无法处理"上次呢？"（4 字符但有明确检索意图）和"你好你好你好你好"（12 字符但无信息量）的边界情况。第二个是**LLM 分类**——让 LLM 判断查询是否值得检索。这在准确性上完胜，但每次查询都要调用 LLM 就违背了"入口轻量"的设计原则——自适应检索的核心价值就是用微秒级的字符串匹配避免毫秒级的检索开销。关键词列表 + 差异化长度阈值的方案，在覆盖率和性能之间找到了最佳平衡点。多语言关键词维护是唯一的持续成本，但相比 LLM 调用的 token 成本，这几乎可以忽略。

**如何判断文本是 CJK 还是英文？** 代码通过检测文本中 CJK Unicode 范围（U+4E00 - U+9FFF 等）字符的比例来判断。当 CJK 字符占比超过一定阈值时，使用 CJK 的最小长度规则。

## 8.2 噪声过滤：出口处的质检员

如果说自适应检索是在入口处拦截无效查询，`noise-filter.ts` 则是在出口处清除低质量结果。即使查询通过了自适应检索的检查，检索管线返回的候选中仍然可能包含噪声记忆——那些不应该被呈现给 LLM 的内容。

```
// 文件: src/noise-filter.ts L10-25
// 噪声过滤器的定位：
// 检索管线的最后几步之一（在 hardMinScore 之后）
// 通过正则表达式模式匹配识别噪声内容
// 匹配到噪声模式的记忆条目会被从结果中移除
```

### Agent 拒绝模式

```
// 文件: src/noise-filter.ts L30-55
// 15 种 agent 拒绝模式，例如：
// - "I cannot help with that"
// - "I'm not able to"
// - "As an AI language model"
// - "I don't have access to"
// - "我无法帮助你"
// - "作为一个 AI"
// ...
```

**为什么要过滤 agent 拒绝内容？** 这些内容是 LLM 自身产生的拒绝回复，不包含任何用户信息。如果它们被存储为"记忆"（可能因为记忆提取逻辑的误判），然后在后续对话中被召回，会导致 LLM 误认为自己之前拒绝过类似请求，从而影响当前回复的质量。过滤掉这些内容是对记忆质量的一种保护。

### 元问题模式

```
// 文件: src/noise-filter.ts L60-90
// 20+ 种元问题模式（含 CJK 变体），例如：
// - "What can you help me with?"
// - "What are your capabilities?"
// - "你能做什么？"
// - "你有什么功能？"
// - "どのようなことができますか？"
// ...
```

元问题是用户询问 AI 能力边界的问题——"你能做什么？"、"你有什么功能？"。这类问题的回答没有长期记忆价值（AI 的能力边界应该由系统提示而非记忆决定），但它们在向量空间中可能与各种话题相关（因为它们本质上是在问"所有话题"），导致被频繁召回。过滤它们可以显著减少噪声干扰。

### 样板文本

```
// 文件: src/noise-filter.ts L95-110
// 样板文本模式：
// - "hi", "hello"（单独出现的问候）
// - "/new", "/clear" 等命令
// - "HEARTBEAT"（某些系统的心跳消息）
// 这些内容如果被误存为记忆，需要在出口处兜底过滤
```

**为什么自适应检索已经跳过了问候语，这里还要再过滤？** 因为自适应检索过滤的是查询端，而噪声过滤过滤的是结果端。一条包含"hello"的记忆可能在自适应检索引入之前就已经被存储了，或者是从对话中提取记忆时的误提取。噪声过滤是最后一道防线，确保无论记忆是何时、如何进入系统的，噪声内容都不会出现在最终结果中。

## 8.3 噪声原型库：基于 Embedding 的自学习噪声识别

`noise-prototypes.ts` 是三层噪声防护中最精巧的一层。与正则表达式的"硬编码"模式不同，它使用 embedding 向量来识别噪声，实现了语言无关的噪声检测，并且能够从运行时数据中自动学习新的噪声模式。

```
// 文件: src/noise-prototypes.ts L10-40
// 噪声原型库的核心思想：
// 1. 预定义一组"种子"噪声原型（15 个）
// 2. 将每个原型编码为 embedding 向量
// 3. 新输入与所有原型计算余弦相似度
// 4. 如果最高相似度超过阈值 → 判定为噪声
// 5. 运行时发现的新噪声 → 加入原型库（自学习）
```

### 15 个种子原型

种子原型覆盖了最常见的噪声类别。它们不是简单的关键词，而是完整的短文本，经过 embedding 编码后在向量空间中"占据"噪声区域。

```
// 文件: src/noise-prototypes.ts L50-100
// 种子原型示例：
// - "Hello, how can I help you today?"（问候）
// - "I'm an AI assistant created by..."（自我介绍）
// - "Sure, I'd be happy to help!"（空洞确认）
// - "Is there anything else you need?"（会话结束语）
// - "Let me think about that..."（填充语）
// ...
// 这些原型的选择标准：
// 高频出现、零信息量、跨语言普遍存在
```

**为什么用 embedding 而不是更多的正则表达式？** 正则表达式是精确匹配工具，它只能匹配预定义的文本模式。但噪声的表达方式是无限多样的——"I'm happy to help" 和 "I'd be glad to assist" 和 "乐意为您效劳" 表达的是同一种噪声语义，但没有任何正则表达式能同时匹配这三种表述。Embedding 方法通过语义空间的距离计算，可以将这些表面不同但语义相近的噪声统一识别。

更重要的是，embedding 方法天然支持多语言。一个英文噪声原型"Hello, how can I help you?"的 embedding 向量，与其中文对应"你好，有什么可以帮你的？"的向量在空间中是接近的（假设使用多语言 embedding 模型）。这意味着我们不需要为每种语言单独维护噪声列表。

### 自学习机制

噪声原型库最令人印象深刻的特性是它的自学习能力：

```
// 文件: src/noise-prototypes.ts L200-250
// 自学习流程：
// 1. 记忆提取模块处理一段对话后得到零条有效记忆
//    （zero-extraction：没有提取出任何值得存储的信息）
// 2. 这段对话文本被视为"疑似噪声"
// 3. 计算其 embedding，与现有原型库去重（阈值 0.95）
// 4. 如果与所有现有原型的最大相似度 < 0.95 → 新噪声模式
// 5. 将其加入原型库
```

**为什么用"零提取"作为学习信号？** 这是一个巧妙的启发式。如果记忆提取模块（通常由 LLM 驱动）分析了一段对话后认为没有任何信息值得存储，那么这段对话很可能确实是噪声。这不是一个完美的信号——有时候 LLM 可能遗漏了有价值的信息——但作为统计意义上的弱信号，它的准确率足够高。

### 容量管理与去重

```
// 文件: src/noise-prototypes.ts L300-340
// 容量控制：
// maxPrototypes: 200
// 当原型数量达到上限时，移除最早加入的原型（FIFO）
//
// 去重阈值：
// dedupThreshold: 0.95
// 新原型与现有原型的余弦相似度 > 0.95 → 视为重复，不加入
```

200 个原型的容量上限是一个平衡点。每次噪声检测需要计算输入与所有原型的余弦相似度，这是 O(n) 的操作。如果原型库无限增长，检测延迟会线性增加。200 个原型在保证覆盖率的同时将检测延迟控制在可接受范围内。

0.95 的去重阈值非常高，这意味着只有几乎完全相同的文本才会被视为重复。这个保守的阈值确保了原型库的多样性——"I'm happy to help you"（相似度可能 0.92）和 "I'm glad to assist"（相似度可能 0.88）会被视为两个不同的原型，分别覆盖向量空间中的不同区域。

## 8.4 三层防护的协同工作

三个模块各司其职，但它们的协同效果远大于各自独立的贡献。

```
// 文件: src/retriever.ts L800-850
// 三层噪声防护在检索管线中的位置：
// [输入] → adaptive-retrieval（入口守门）
//        ↓ 通过
// [检索管线 Step 1-7]
//        ↓
// noise-filter（出口质检，正则匹配）
//        ↓
// noise-prototypes（embedding 语义检测）
//        ↓
// [输出] 干净的检索结果
```

**第一层（自适应检索）** 在入口处拦截了大约 30-40% 的无效查询，节省了完整检索管线的计算成本。它的特点是快速——只需要几次字符串匹配和长度检查。

**第二层（噪声过滤器）** 在出口处清除匹配已知噪声模式的结果。正则表达式的匹配速度极快（微秒级），即使结果集中有数十条候选也不会引入可感知的延迟。它的覆盖范围有限（只能匹配预定义的模式），但对已知噪声的清除是确定性的。

**第三层（噪声原型库）** 补全了前两层的盲区。自适应检索只过滤查询端，不过滤结果端。正则过滤只匹配已知模式，无法应对新出现的噪声变体。噪声原型库通过 embedding 语义匹配覆盖了无限多的噪声表述形式，并且通过自学习机制不断扩展覆盖范围。

三层防护的叠加确保了 memory-lancedb-pro 的检索结果具有高信噪比。在生产环境中，这直接影响了 LLM 利用记忆的效果——干净的记忆让 LLM 能够准确回忆历史信息，而噪声记忆则会导致混乱、重复和幻觉。

## 本章小结

自适应检索、噪声过滤和噪声原型库构成了 memory-lancedb-pro 的三层噪声防护体系。自适应检索（`adaptive-retrieval.ts`）在入口处通过 skip/force/normal 三路分类拦截无效查询，其多语言关键词和差异化长度阈值（英文 15 字符、CJK 6 字符）体现了对语言特性的深入理解。噪声过滤器（`noise-filter.ts`）在出口处用正则表达式清除 agent 拒绝、元问题和样板文本三类已知噪声。噪声原型库（`noise-prototypes.ts`）是最精巧的一层：15 个种子原型通过 embedding 实现语言无关的噪声检测，零提取信号驱动的自学习机制使其覆盖范围随运行时间自动扩展，200 上限和 0.95 去重阈值平衡了覆盖率和性能。三层防护从快到精、从入口到出口、从硬编码到自学习，层层递进，共同保障了检索结果的信噪比。
