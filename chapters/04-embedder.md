# 第四章：向量嵌入与分块策略

> 向量嵌入是连接人类语言与机器理解的桥梁。一个记忆系统的检索质量，很大程度上取决于嵌入的质量。memory-lancedb-pro 在嵌入层投入了 1,000 行代码，不是因为嵌入本身复杂，而是因为在生产环境中让嵌入稳定、高效、可靠地工作，需要解决大量工程问题。

## 4.1 多提供商抽象层

`embedder.ts`（800 行）支持 5 种嵌入模型提供商——Jina、OpenAI、Voyage、Gemini、Ollama，每种都有自己的 API 格式、认证方式和响应结构。

```
// 文件: src/embedder.ts L20-60
// 所有提供商都实现了 OpenAI 兼容的嵌入 API 格式。
// 这不是巧合，而是刻意的架构选择——OpenAI 的
// /v1/embeddings 端点已成为事实上的行业标准。
// 对于不完全兼容的提供商（如 Ollama 的本地模式），
// embedder 内部做了适配层转换。
```

为什么不直接使用各提供商的官方 SDK？因为引入 5 个 SDK 会极大膨胀插件的依赖树，而嵌入 API 的调用逻辑非常简单——构建一个 HTTP POST 请求，发送文本，接收向量。自行实现 HTTP 调用让 memory-lancedb-pro 保持了轻量级的依赖结构，也避免了 SDK 版本升级带来的兼容性问题。

### 核心流程

```
embedder.ts 嵌入生成流程:

+--------+    +----------+    +---------+    +----------+    +--------+
| 输入    +--->+ LRU 缓存  +--->+ 选择    +--->+ HTTP     +--->+ 向量   |
| 文本    |    | 命中检查   |    | Provider|    | POST 请求 |    | 返回   |
+--------+    +----+-----+    +---------+    +-----+----+    +--------+
                   |                               |
              [命中] 直接返回                  [失败] 轮换 Key
                                              401 → 标记 60min
                                              429 → 标记 60s

chunker.ts 长文本处理流程:
+--------+    +-----------+    +---------+    +----------+    +--------+
| 长文本   +--->+ 句子边界   +--->+ 贪心合并 +--->+ 并行嵌入  +--->+ 向量   |
| 输入    |    | 识别切分   |    | + 重叠   |    | (信号量)  |    | 平均   |
+--------+    +-----------+    +---------+    +----------+    +--------+
```

```
// 文件: src/embedder.ts L70-100
// 每个提供商有预配置的默认 baseUrl 和认证 header 格式。
// Jina、OpenAI、Voyage 都使用 Bearer token，
// 但 header name 和格式可能微妙不同。
// 这些差异都被封装在 embedder 内部，对外只暴露统一接口。
```

## 4.2 任务感知嵌入（Task-Aware Embedding）

这是 `embedder.ts` 中最精妙的设计之一。在信息检索领域，**查询文本和文档文本的最优嵌入方式是不同的**。查询通常简短且意图明确；被检索的文档通常较长且信息密集。

```
// 文件: src/embedder.ts L120-170
// embed(text: string, task: 'retrieval.query' | 'retrieval.passage')
//
// 'retrieval.query' 时强调意图捕捉和语义概括。
// 'retrieval.passage' 时强调内容覆盖和细节保留。
// Jina v3 和 Voyage 原生支持 task 参数；
// 对于不支持的模型，embedder 静默忽略 task 参数。
```

为什么这个区分如此重要？想象用户问"我之前用过什么数据库？"，记忆库中有"用户在项目中使用了 PostgreSQL 和 Redis"。如果查询嵌入被优化为捕捉意图（"数据库"意味着想知道具体产品），而文档嵌入被优化为保留细节（"PostgreSQL" 和 "Redis" 被标记为数据库类实体），匹配质量就会显著提升。

### 设计取舍

**任务感知嵌入**的替代方案是对所有文本使用统一的嵌入方式（不区分 query 和 passage）。统一方式实现简单且所有模型都支持，但在非对称检索场景（短查询 vs 长文档）中会损失 5-15% 的检索精度。memory-lancedb-pro 选择了任务感知方案，对不支持 task 参数的模型静默忽略，实现了"有则更好、无则不差"的渐进增强。**自行实现 HTTP 调用 vs 使用官方 SDK** 是另一个关键取舍：5 个 SDK 会引入数十个传递依赖，增加安装时间和潜在的版本冲突；而嵌入 API 的调用模式极其简单（一个 POST 请求），自行实现的维护成本远低于管理 5 个 SDK 的兼容性矩阵。

## 4.3 LRU 缓存与 30 分钟 TTL

嵌入 API 调用有两个成本：延迟（50-200 毫秒）和费用（按 token 计费）。在记忆系统中，很多文本会被重复嵌入——同一个查询可能被多次使用，同一条记忆在去重阶段可能被反复嵌入比较。

```
// 文件: src/embedder.ts L200-260
// LRU 缓存设计：
// - 缓存键：hash(text + model + task)
// - 缓存值：float32[] 向量
// - TTL：30 分钟
//
// 为什么是 30 分钟？1. 内存占用——每个 1024 维向量约 4KB，
// 作为插件不应无限制占用宿主进程内存。
// 2. 模型一致性——开发阶段用户可能切换模型，
// TTL 确保缓存不会长时间持有过时结果。
```

缓存键包含 `model` 和 `task` 参数——同一段文本在不同模型下的嵌入完全不同，在不同 task 下也可能不同。如果缓存键只用文本内容，切换模型时会返回错误的缓存结果。

## 4.4 轮询密钥轮换与故障转移

单个 API Key 有速率限制，高频使用场景下容易触达上限。memory-lancedb-pro 支持配置多个 API Key，通过轮询（Round-Robin）策略分散负载。

```
// 文件: src/embedder.ts L280-340
// getNextKey(): string {
//   const key = this.keys[this.keyIndex % this.keys.length];
//   this.keyIndex++;
//   return key;
// }
```

更关键的是故障转移逻辑：

```
// 文件: src/embedder.ts L350-410
// 当请求返回 401 或 429 时，将 key 标记为"暂时不可用"，
// 自动切换到下一个 key 重试。
// - 401 错误：标记 60 分钟（可能是 key 被吊销）
// - 429 错误：标记 60 秒（速率限制通常很快恢复）
// 如果所有 key 都不可用，重置所有标记并用第一个 key 重试。
```

这段逻辑只有几十行代码，但生产价值巨大。没有它，一个 API Key 的临时失效就会导致整个记忆系统停摆。

## 4.5 自动维度检测

不同模型输出不同维度的向量——OpenAI text-embedding-3-small 是 1536 维，Jina v3 是 1024 维。创建 LanceDB 表时需要知道维度才能正确定义 schema。

```
// 文件: src/embedder.ts L430-490
// 系统内置 25+ 常见模型的维度映射表。
// const KNOWN_DIMENSIONS: Record<string, number> = {
//   'text-embedding-3-small': 1536,
//   'jina-embeddings-v3': 1024,
//   'voyage-3': 1024,
//   'text-embedding-004': 768,  // Gemini
//   // ... 25+ 模型
// };
// 对于未知模型，发送探测请求从响应中获取实际维度。
```

为什么不让用户手动指定维度？因为这是容易出错的配置项——记错维度、切换模型时忘记更新。自动检测消除了这类错误，同时配置中仍保留 `dimensions` 字段允许显式覆盖。

## 4.6 语义分块策略

`chunker.ts`（200+ 行）处理长文本切分。最朴素的方式是按固定字符数截断，但这经常在句子甚至词语中间断开。memory-lancedb-pro 采用**语义感知的句子边界切分**。

```
// 文件: src/chunker.ts L20-60
// 切分算法：
// 1. 使用正则表达式识别句子边界（区分 "Mr." 和句号等）
// 2. 贪心地将句子合并为 chunk，直到达到目标长度
// 3. 新 chunk 的开头包含前一个 chunk 的最后 N 个句子（重叠）
```

**为什么需要重叠？** 考虑："用户提到他去年搬到了上海。他说上海的天气让他很不适应。"无重叠地切分后，第二个 chunk 失去了"他"的指代上下文。重叠确保每个 chunk 携带足够的上下文信息。

```
// 文件: src/chunker.ts L70-110
// minChunkLength 防止产生过短的 chunk。
// 过短文本（如"好的。"）嵌入后的向量信息密度很低，
// 会在检索时产生大量误匹配——短文本的嵌入向量
// 往往落在高维空间的"公共区域"，
// 与很多查询都有中等相似度但没有真正的相关性。
```

## 4.8 嵌入层的优雅降级

嵌入层有一个**故意缺失**的降级路径——跨提供商降级：

```
// 文件: src/embedder.ts L550-600
// 嵌入层没有"Jina 不可用时自动切换到 OpenAI"的设计。
// 不同提供商的嵌入向量处于完全不同的向量空间，
// 混用会导致检索结果完全不可靠。
// 与其给出错误的结果，不如明确地失败。
```

这体现了工程成熟度：所有记忆和查询必须使用同一个嵌入模型，否则向量空间不一致，相似度计算毫无意义。悄悄切换提供商比不返回结果更糟糕——用户会认为系统正常工作，但实际得到的是垃圾数据。

## 4.9 embedder 与 chunker 的协作模式

两者的协作关系是单向依赖：chunker 依赖 embedder，但 embedder 不知道 chunker 的存在。

```
// 文件: src/embedder.ts L620-650
// 调用链路：
// index.ts → 判断文本长度是否需要分块
//   → 需要：chunker.chunk(text)，对每个 chunk 调用 embedder.embed()
//   → 不需要：直接 embedder.embed(text)
// 分块阈值由配置决定（默认约 512 tokens）。
```

embedder 保持单一职责——"给我文本，还你向量"。分块逻辑、向量聚合在 chunker 中处理，何时分块由编排层决定。每个组件只关心自己的领域，通过清晰的接口协作。

## 本章小结

`embedder.ts` 和 `chunker.ts` 共同构成了 memory-lancedb-pro 的嵌入层，总计约 1,000 行代码。嵌入层通过 OpenAI 兼容的 API 格式抽象了 5 种提供商，通过任务感知嵌入提升检索匹配质量，通过 LRU 缓存和轮询密钥轮换解决了性能和可靠性问题，通过自动维度检测降低了配置出错的可能性。chunker 采用语义感知的句子边界切分和可配置重叠策略，结合并行嵌入与向量平均，在保持语义完整性的同时高效处理长文本。嵌入层故意不支持跨提供商降级，这个看似"缺陷"的设计决策，恰恰体现了对向量空间一致性这一核心约束的深刻理解。
