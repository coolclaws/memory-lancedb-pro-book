# 第九章 检索评分体系

> 搜索引擎的灵魂不在于找到什么，而在于如何排序。memory-lancedb-pro 的检索评分体系像一位经验丰富的图书管理员——不仅知道哪些书与你的问题相关，还能根据书的新旧程度、你的借阅历史、以及避免推荐重复内容等多重维度，为你精心排列出最终的推荐列表。本章将深入剖析这套后融合评分管线（post-fusion scoring pipeline）的每一个环节，揭示每个设计决策背后的 "为什么"。

## 9.1 评分管线总览

在 `retriever.ts` 中，一条记忆从向量检索返回到最终呈现给用户，需要经过四道评分关卡：

1. **长度归一化**（Length Normalization）
2. **生命周期衰减增强**（Lifecycle Decay Boost）
3. **硬性最低分过滤**（Hard Min Score Filter）
4. **MMR 多样性去重**（MMR Diversity）
5. **噪声过滤**（Noise Filter）

这五层并非随意堆叠，而是按照严格的逻辑顺序排列：先修正原始分数的偏差，再融入时间维度的权重，然后裁剪低质量结果，接着消除冗余，最后做正则表达式级别的噪声清理。这种分层设计的核心理念是 **关注点分离**——每一层只解决一个特定问题，层与层之间通过乘法或过滤操作组合。

```typescript
// 文件: src/retriever.ts L45-52
// 伪代码展示评分管线的整体流程
for (const candidate of rawResults) {
  candidate.score *= lengthNormFactor(candidate);
  candidate.score *= decayEngine.score(candidate.memory).composite;
  if (candidate.score < 0.35) continue; // hard min
  candidates.push(candidate);
}
const selected = applyMMR(candidates, 0.85);
const final = applyNoiseFilter(selected);
```

## 9.2 长度归一化：为什么用对数，为什么锚点是 500 字符

向量相似度搜索存在一个众所周知的偏差：较长的文本往往包含更多的语义信号，因此在余弦相似度计算中容易获得更高的分数。如果不加以修正，系统会系统性地偏好长文本，而那些精炼但高度相关的短记忆则会被埋没。

```typescript
// 文件: src/retriever.ts L78-82
const lengthNormAnchor = 500;
score *= 1 / (1 + Math.log2(charLen / lengthNormAnchor));
```

这个公式有三个关键设计决策值得深究：

**为什么用 `log2` 而不是线性除法？** 如果使用线性归一化（例如 `score /= charLen`），那么一段 2000 字符的记忆会被惩罚到 500 字符记忆的四分之一，这过于激进。对数函数的增长速度远慢于线性函数——当字符数从 500 增长到 2000 时，`log2(2000/500) = log2(4) = 2`，归一化因子为 `1/3`，惩罚力度温和得多。对数的本质特性是：**对初始的长度差异敏感，但随着文本越来越长，额外的长度带来的惩罚递减**。这恰好符合直觉——一段 500 字和 1000 字的记忆在信息密度上差异显著，但 5000 字和 5500 字的记忆差异就不那么重要了。

**为什么锚点选择 500 字符？** 锚点 `lengthNormAnchor` 的含义是：当文本恰好为 500 字符时，归一化因子为 `1 / (1 + log2(1)) = 1 / (1 + 0) = 1`，即不做任何调整。500 字符大约对应 2-3 个自然段落，这是大多数记忆条目的 "典型长度"。选择这个值意味着系统将 500 字符视为基准线——比它短的记忆会获得微小的加分（因为 `log2` 的参数小于 1 时为负值，导致分母小于 1），比它长的记忆则受到递增的惩罚。这个锚点值不是凭空选择的，而是基于记忆系统中条目长度分布的中位数附近确定的。

**为什么分母是 `1 + log2(...)` 而不是直接用 `log2(...)`？** 加 1 保证了当 `charLen = lengthNormAnchor` 时因子恰好为 1（中性值），同时避免了 `charLen < lengthNormAnchor` 时分母趋近于零导致分数爆炸的问题。这是一个防御性设计，确保归一化因子始终在合理范围内。

## 9.3 生命周期衰减增强：连接时间维度

原始的向量相似度只衡量语义相关性，完全忽略了时间维度。但在记忆系统中，一条昨天创建的记忆和一条三个月前的记忆，即使语义相似度相同，其实际价值可能天差地别。

```typescript
// 文件: src/retriever.ts L85-87
score *= decayEngine.score(memory).composite;
```

这行代码将衰减引擎（详见第十一章）计算的复合分数直接乘到检索分数上。这里采用乘法而非加法是深思熟虑的选择：**乘法意味着即使语义相关性很高，如果记忆已经严重衰减，最终分数也会被大幅压低**。相反，如果用加法（`score += decayScore`），一条语义不太相关但非常新鲜的记忆可能会排到前面，这不符合 "相关性优先" 的设计哲学。

衰减引擎返回的 `composite` 分数是一个 0 到 1 之间的值，综合了新近度（recency）、访问频率（frequency）和内在重要性（intrinsic）三个维度。乘法操作的另一个好处是它天然地保持了分数的有界性——如果原始分数在 [0, 1] 范围内，乘以一个 [0, 1] 的衰减分数后仍然在 [0, 1] 范围内，无需额外的归一化步骤。

## 9.4 硬性最低分过滤：为什么阈值是 0.35

经过长度归一化和衰减增强后，系统需要一个 "质量底线" 来裁剪掉那些虽然被向量搜索返回但实际上并不够好的结果。

```typescript
// 文件: src/retriever.ts L90-91
if (candidate.score < 0.35) continue; // hard min score filter
```

**为什么是 0.35 而不是 0.5 或 0.2？** 这个阈值的选择需要平衡两个风险：阈值太高会导致召回率不足（false negatives），用户需要的记忆被过滤掉；阈值太低则会引入太多噪声（false positives），降低检索质量。0.35 这个值意味着经过长度归一化和衰减增强后，一条记忆的有效分数至少需要达到满分的三分之一强才能通过。

考虑一个具体场景：一条语义相似度为 0.6 的记忆（中等相关），如果它的衰减复合分数也是 0.6（中等新鲜），那么最终分数约为 `0.6 × 0.6 = 0.36`，刚好能通过过滤。这意味着 **系统允许 "中等相关 + 中等新鲜" 的记忆通过，但不允许 "低相关 + 低新鲜" 的记忆通过**——这是一个合理的质量门槛。

值得注意的是，这是一个 "硬性" 过滤器，没有模糊边界。这种设计简单直接，避免了引入额外的超参数（如软过滤的温度参数），降低了调优复杂度。

## 9.5 MMR 多样性：为什么相似度阈值是 0.85

Maximal Marginal Relevance（MMR）是信息检索领域的经典技术，用于在保持相关性的同时引入多样性。memory-lancedb-pro 的实现聚焦于一个核心问题：**当多条记忆说的是几乎相同的事情时，只保留最好的那一条**。

```typescript
// 文件: src/retriever.ts L105-115
// MMR 多样性过滤
for (const candidate of sortedCandidates) {
  let dominated = false;
  for (const selected of result) {
    if (cosineSimilarity(candidate.embedding, selected.embedding) > 0.85) {
      candidate.score *= 0.5; // demote score
      dominated = true;
      break;
    }
  }
  // ...
}
```

**为什么阈值是 0.85 而不是更低？** 0.85 的余弦相似度在高维向量空间中意味着两个向量几乎指向同一方向——它们描述的是 "几乎相同" 的语义内容。如果阈值设得更低（比如 0.7），系统会过度去重，可能将仅仅 "相关" 但并非 "重复" 的记忆也降权。0.85 是一个保守的选择，确保只有真正的近义重复才会被处理。

**为什么是降权（demote）而不是直接删除？** 注意代码中的处理方式是 `score *= 0.5`，将分数减半而非完全移除。这种 "软去重" 的设计保留了一个重要的安全网：如果降权后的记忆仍然高于 0.35 的硬性最低分，它仍然有机会出现在结果中。在某些场景下，用户可能确实需要看到同一主题的多个视角，完全删除会丢失信息。

**与已选集合比较的贪心策略。** MMR 的实现采用贪心算法——每个候选记忆只与已经被选中的记忆比较，而不是与所有其他候选记忆比较。这意味着选择的顺序很重要：分数最高的记忆先被选入，后续的记忆如果与它过于相似就会被降权。这种贪心策略的时间复杂度是 O(n×k)（n 为候选数，k 为已选数），远优于全局最优解的 O(n²)，在实际使用中效果足够好。

## 9.6 噪声过滤：最后的防线

```typescript
// 文件: src/retriever.ts L130-135
// 基于正则表达式的噪声过滤
const final = applyNoiseFilter(selected);
```

噪声过滤是管线的最后一道关卡，使用正则表达式匹配来识别和移除那些在语义上可能得分不低、但实际内容是噪声的结果。典型的噪声包括：纯格式文本、自动生成的模板内容、系统日志片段等。这些内容可能在向量空间中与查询有一定的相似度（因为它们包含相关的关键词），但对用户来说毫无价值。

将噪声过滤放在管线最后而非最前面，是因为正则表达式匹配虽然计算成本低，但需要处理的模式可能很多。放在最后意味着它只需要处理经过前面四层筛选后的少量结果，而非全部候选。

## 9.7 完整评分公式

将所有层次组合起来，一条记忆的最终评分可以表示为：

```
finalScore = vectorSimilarity
           × (1 / (1 + log2(charLen / 500)))
           × decayComposite
           × (isDuplicate ? 0.5 : 1.0)
```

其中 `finalScore >= 0.35` 且通过噪声正则过滤才能最终返回。

这个公式的每一个乘法因子都对应一个独立的关注点：语义相关性、长度偏差修正、时间衰减、多样性。乘法的组合方式意味着任何一个维度的严重不足都会拉低总分——这是一种 "木桶效应" 的设计，确保最终返回的记忆在所有维度上都达到基本水平。

## 本章小结

本章详细剖析了 memory-lancedb-pro 的检索评分管线。该管线由五个层次组成，每一层解决一个特定问题：长度归一化修正向量搜索对长文本的偏好，基于对数函数实现温和的惩罚曲线；生命周期衰减增强将时间维度融入评分，通过乘法确保相关性仍然是第一优先级；0.35 的硬性最低分在 "中等相关 + 中等新鲜" 的水平画了一条质量底线；MMR 多样性以 0.85 的高阈值保守地识别近义重复，用降权而非删除的方式保留安全网；噪声过滤作为最后的防线清理格式噪声。这五层通过乘法和过滤操作优雅地组合，形成了一个既强大又可调的评分体系。理解这套管线，是理解整个检索系统为什么能返回 "恰到好处" 的结果的关键。
