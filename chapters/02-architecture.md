# 第二章：架构全景与模块依赖

> 好的架构不是让系统跑起来，而是让系统在出问题时依然能跑下去。memory-lancedb-pro 的分层架构和优雅降级设计，正是这一理念的最佳实践。

## 2.1 分层架构概览

memory-lancedb-pro 的整体架构可以抽象为四个层次，从底向上依次为：**存储层（Storage）**、**检索层（Retrieval）**、**提取层（Extraction）**、**集成层（Integration）**。这种分层不是为了美观，而是出于一个务实的工程考量——每一层都应该能够独立演进，并且在上层出现故障时不会拖垮下层。

```
┌─────────────────────────────────────────────┐
│  集成层 Integration                          │
│  index.ts / tools.ts / cli.ts               │
├─────────────────────────────────────────────┤
│  提取层 Extraction                           │
│  smart-extractor.ts / extraction-prompts.ts │
│  memory-categories.ts / smart-metadata.ts   │
├─────────────────────────────────────────────┤
│  检索层 Retrieval                            │
│  retriever.ts / adaptive-retrieval.ts       │
│  noise-filter.ts / noise-prototypes.ts      │
│  decay-engine.ts / tier-manager.ts          │
├─────────────────────────────────────────────┤
│  存储层 Storage                              │
│  store.ts / embedder.ts / chunker.ts        │
│  scopes.ts / workspace-boundary.ts          │
└─────────────────────────────────────────────┘
```

为什么把作用域（scopes）放在存储层而不是检索层？因为作用域本质上是数据的访问边界——它决定的是"哪些数据对当前查询可见"，这是一个存储级的关注点。在数据写入时就打上 scope 标签，远比在检索时动态过滤更高效、更安全。

```
// 文件: src/scopes.ts L50-80
// scope 标签在记忆写入时即被确定，存储为记忆记录的
// 一个字段。这意味着作用域隔离是存储级别的"硬隔离"，
// 而非检索级别的"软过滤"。即使检索层出现 bug，
// 不同作用域的数据也不会泄露。
```

## 2.2 四大子系统详解

**存储子系统** 的核心是 `store.ts`（700 行），封装了与 LanceDB 的所有交互。LanceDB 是嵌入式的列式向量数据库，不需要独立服务进程——这是选择它的关键原因。`embedder.ts`（800 行）抽象了 5 种嵌入模型提供商，`chunker.ts`（200+ 行）处理长文本的语义感知分块。

```
// 文件: src/store.ts L30-60
// LanceDB 连接采用懒初始化模式。为什么不在插件启动时
// 就建立连接？因为不是每次对话都需要访问记忆数据库。
// 懒初始化避免了不必要的 I/O 开销，同时也简化了
// 错误处理——连接失败只会影响实际需要记忆的操作。
```

**检索子系统** 以 `retriever.ts`（900 行）为核心，实现了完整的检索流水线。流水线从查询嵌入开始，经过向量 ANN 搜索和 BM25 全文搜索两条并行路径，通过加权融合合并结果，随后依次经过 Cross-Encoder 重排序、长度归一化、衰减加成、硬性最低分过滤（0.35）、噪声过滤、MMR 多样性去重。`decay-engine.ts`（230 行）和 `tier-manager.ts`（190 行）共同构成生命周期管理模块，使用 Weibull 衰减模型和三级记忆层（Core / Working / Peripheral）。

```
// 文件: src/tier-manager.ts L20-50
// 三级记忆层借鉴了操作系统中的多级缓存思想。
// Core 层类似 L1 缓存，衰减慢（beta=0.8），适合核心偏好；
// Working 层衰减适中（beta=1.0）；
// Peripheral 层衰减最快（beta=1.3），是淘汰候选区。
```

**提取子系统** 的核心是 `smart-extractor.ts`（700 行），依赖 LLM 从对话中提取值得记忆的信息。两阶段去重是其最精妙的设计：先通过向量预过滤（阈值 0.7）快速排除重复内容，再交由 LLM 进行语义决策。`smart-metadata.ts`（400 行）实现了 L0/L1/L2 三层元数据结构，让简单查询只需读取 L0，复杂分析可以深入到 L2。

**集成子系统** 包括 `index.ts`（3,500+ 行）作为编排中心，`tools.ts`（800 行）定义 Agent 工具接口，`cli.ts`（1,100+ 行）提供命令行管理能力，以及反射子系统（`reflection-*.ts`，4,000+ 行）负责会话级记忆反思。

## 2.3 index.ts 的编排角色

`index.ts` 不仅是 OpenClaw 插件的注册入口，更是所有子系统的编排者：

```
// 文件: index.ts L300-380
// 典型的记忆写入流程：
// 1. 接收对话消息（来自 OpenClaw 的 hook 回调）
// 2. 调用 smart-extractor 从对话中提取记忆候选项
// 3. 对每个候选项，通过 scopes 确定目标作用域
// 4. 调用 embedder 生成向量嵌入
// 5. 调用 store 写入 LanceDB
// 6. 通过 access-tracker 记录初始访问信息
// 7. 触发 tier-manager 评估是否需要层级调整
```

index.ts 之所以有 3,500 行，是因为它承担了大量胶水逻辑——参数校验、错误处理、日志记录、降级判断。这些代码不"酷"，但却是生产级软件不可或缺的部分。

## 2.4 初始化顺序

插件的初始化顺序反映了模块间的依赖关系：

```
// 文件: index.ts L100-180
// 1. 配置解析与验证 → 确保所有参数合法
// 2. 嵌入模型初始化 → 验证 API 连通性，检测向量维度
// 3. LanceDB 连接（懒初始化，此时仅准备连接参数）
// 4. 作用域管理器初始化 → 注册默认作用域规则
// 5. 检索器 / 提取器 / 生命周期管理器初始化
// 6. 工具注册 → 向 OpenClaw 注册 Agent 工具
// 7. 反射子系统初始化 → 根据 sessionStrategy 配置会话策略
```

为什么嵌入模型要在 LanceDB 之前初始化？因为如果嵌入服务不可用，后续的所有操作都没有意义——既无法写入也无法检索。尽早发现嵌入层的问题，可以在启动阶段就给出明确的错误提示。

## 2.5 优雅降级的系统性设计

优雅降级不是"加几个 try-catch"就能实现的。memory-lancedb-pro 的降级设计是系统性的：

```
// 文件: index.ts L400-450
// 降级链遵循一个原则：每个功能都有至少一个降级方案。
// LLM 提取 → 正则提取 → 跳过提取（仍可手动存储）
// Cross-Encoder 重排序 → 余弦相似度排序 → 原始分数排序
// OAuth 认证 → API Key 认证 → 匿名模式（受限功能）
// FTS 全文搜索 → 纯向量搜索 → 关键词匹配降级
```

这种设计源于一个现实：生产环境中外部依赖的可用性永远无法保证 100%。用户可能不会注意到检索结果从"优秀"变成"良好"，但一定会注意到系统完全不可用。每个子系统都暴露了"能力状态"接口，编排层据此动态调整处理流程。

## 2.6 模块依赖关系

关键的依赖链如下：

- `retriever.ts` 依赖 `store.ts`、`embedder.ts`、`decay-engine.ts`、`noise-filter.ts`
- `smart-extractor.ts` 依赖 `llm-client.ts`、`embedder.ts`、`store.ts`、`memory-categories.ts`
- `tier-manager.ts` 依赖 `access-tracker.ts`、`decay-engine.ts`、`store.ts`

```
// 文件: src/llm-client.ts L20-50
// LLM 客户端被 smart-extractor 和 reflection 子系统共用。
// 它封装了请求构建、超时控制、重试逻辑和错误处理，
// 避免了 LLM 调用逻辑在多处重复实现。
```

一个值得注意的设计：`store.ts` 不依赖 `embedder.ts`。存储层只负责存储和检索原始数据（包括预计算的向量），嵌入的生成由上层负责。这种解耦确保了存储层的纯粹性——它不需要关心向量是怎么来的，只需要知道如何高效地存储和查询它们。

## 本章小结

memory-lancedb-pro 采用四层架构（存储、检索、提取、集成），由 index.ts 作为编排中心协调各子系统的工作。初始化顺序严格遵循依赖关系，嵌入层优先于存储层初始化以实现快速失败。优雅降级是系统性设计而非临时补丁，每个外部依赖都有明确的降级路径。模块间通过接口抽象降低耦合，存储层与嵌入层的解耦尤其值得借鉴。理解了这个架构全景，我们就有了深入各个子系统实现细节的基础。
